# Agentic Design Principles (AGENTS_CN.md)

本文件是 **Synapse** 平台内所有关于大模型（LLM）行为设计、提示词架构和图流转控制的“宪法”。
如果你是后续接手平台的 AI 研发或架构师，在设计任何新的节点流转、Prompt 模板或 MCP 工具时，请严格遵守以下原则。

## 1. 核心范式：Flow Engineering > Prompt Engineering

我们彻底摒弃了传统 Prompt Skill（如所谓的 Panic Analyzer）那种“几千字自然语言约束、全程任由大模型调用工具发散”的做法。因为大模型极其昂贵、缓慢，且容易产生致命的幻觉。

**在 Synapse 中：**
* 大模型是**节点（Node）**，而不是**调度器（Scheduler）**。
* 调度器是坚固的 Golang 图执行引擎（DAG Engine）。

## 2. 节点二元论：硬节点与软节点

我们在画布上定义的所有节点，被严格划分为两类：

### 2.1 硬节点 (Hard Nodes / 肌肉与手脚)
* **本质**：不调用任何大模型，100% 确定性的代码执行。
* **涵盖**：执行 Bash 脚本、发起 MCP HTTP 查询（查 DB/ES日志）、执行预置的 Python 检查脚本等。
* **规则**：遇到确定的告警特征（如 TraceID），引擎立刻并发启动对应的硬节点拉取数据。此过程 Token 消耗为 0，耗时为毫秒到秒级。必须优先让硬节点把所有事实（Facts）拉取完毕。

### 2.2 软节点 (Soft Nodes / 大脑与感官)
* **本质**：调用大模型 API (如 Claude/GPT)，执行复杂的模式识别与推理。
* **触发时机**：只有在硬节点执行完毕，收集到足够的确凿证据时才触发。
* **提示词设计原则**：软节点的 Prompt 永远是 **“基于以下已收集的事实，进行逻辑推理和根因分析”**，而不是“你觉得还需要查什么，请调用工具去查”。
* **结构化输出**：软节点必须强制输出标准化的 JSON（如结论类型、根因详情、是否有缺失数据等），以便引擎后续进行条件路由。

### 2.3 自动化边界：WebMCP 与大模型的严格职责划分
在引入 **WebMCP** 实现 UI 前端自动化测试时，我们必须严格划清机器（浏览器）与 AI（大模型）的边界，杜绝“用大炮打蚊子”。
* **❌ 严禁使用大模型做的（执行层）**：在页面上寻找元素、识别按钮、输入文字、切换标签页。这些动作极度脆弱且昂贵，必须通过拦截底层 WebMCP 语义工具调用的方式，固化为 100% 确定性的**硬节点 (Hard Node)**，由引擎通过 CDP 协议光速重放。
* **✅ 必须使用大模型做的（决策与拓展层）**：
  1. **复杂断言验证**：作为**软节点**挂载在流程末尾，接收 WebMCP 返回的页面状态数据（而非截图），分析“本次业务流程（如发货）是否成功”。
  2. **留痕与总结**：在流程执行报错时，分析并归因，撰写测试报告或排查文档。
  3. **动态遍历路由**：作为 `Router Node`，在某些业务分支处，根据当前系统的反馈，动态生成不同的入参（如尝试不同的审批流、遍历不同站点），驱动下方的 WebMCP 节点执行。

## 3. 有限状态机与循环熔断 (Statechart & Circuit Breaker)

传统的 DAG 是单向流动的，但真实的排查问题往往需要“线索不足 -> 反思 -> 更换关键字重新查”的循环。

**Synapse 的图循环控制规范：**
1. **统一全局状态池 (`Global State`)**：在整个图流转期间，维护一个统一的状态对象。软节点（LLM）的输出（比如：发现最初的 TraceID 查询日志不全，需要根据新关联的 OrderID 查）会被追加（Append/Patch）到 `State` 中。
2. **基于 State 的条件路由 (Router Edges)**：前端导出的 JSON 必须指明：“如果软节点输出 `missing_info == true`，则连线指回前置的硬节点”。当前置硬节点再次触发时，它将从更新后的 `State` 中读取新的查询关键字（OrderID），有效防止使用相同的条件陷入死循环。
3. **强制熔断 (Circuit Breaker)**：底层引擎必须为每一个拥有循环回路的图隐式注入 `loop_count`。如果某段循环执行超过 N 次（如 3 次），必须强制拉出循环，自动跳转进入 **“人工干预节点 (Human-in-the-loop)”**，挂起状态机等待 SRE 介入，防止大模型陷入“幻觉死锁”烧毁账单。

## 4. 双循环记忆库 (The Shadow Memory Agent)

借鉴自顶级开源工程的 RAG 闭环设计，Synapse 绝不允许在主流程的 System Prompt 中“硬编码”填鸭所有的排查经验。

### 4.1 主动记忆（闭环萃取）
在一次排查结束、并且结论得到人工验证后，引擎会拉起一个完全独立的后台**影子智能体 (Shadow Extraction Agent)**。
* **它的职责**：阅读本次排查完整的“全局状态 State（经历的路径、获取的特征、人工的修改）”，剥离无关杂讯，提取出纯粹的“告警特征与排查路径模板”。
* **它的权限**：极度受限（沙盒环境）。它只能使用写权限的 MCP 工具将这段结构化经验（Markdown/JSON）存入指定的知识库（Vector DB）或记忆文件夹。禁止它跑任何外网或环境查询。

### 4.2 被动召回（执行预加载）
当新的告警事件触发排查图的起点时，系统会在第一个前置硬节点中，加入一个并行的“经验召回节点”。
利用相似度检索（基于日志报错堆栈特征或关键字），它会瞬间从向量库中调出那条曾经写入的“排查拓扑路径和注意事项”，作为附加的 Context（外挂大脑），一起喂给后续的分析节点。这就是**特征精准路由**的落地，彻底解决复杂业务经验复用时大模型“记忆遗忘”的问题。
